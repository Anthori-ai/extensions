# Filesystem

First-party project extension that exposes filesystem controls through the sandboxed executable host.

`Create` returns `fileHash` for created files so callers can immediately make guarded `Read`, `Read Lines`, `Write`, or `Write Lines` calls. Directory creation returns an empty `fileHash`. `Info` returns file or directory metadata; for files, it includes the same opaque file version token.

`Read` normally requires the expected current `fileHash`, then returns decoded file content for a byte range selected by `offset` and `limit`, plus `fileByteCount`, `bytesRead`, the current `fileHash`, and an optional byte `rangeKey` formatted as an opaque base62 token when bytes are returned. Get the initial `fileHash` from `Create` or `Info`, then use the latest `fileHash` returned by creates, reads, and writes. For files that change too quickly to hash first, such as active logs, pass `dangerous: true` instead of `fileHash` to bypass stale-read detection. Pull-based Read sends file data in `chunk.content`; the terminal `end` event carries the read summary fields top-level and does not repeat content.

`Write` uses byte `rangeKey` values returned by `Read` for `replace`, `insert`, and `delete`, so raw byte edits target exactly the read range and fail if that range changed. `insert` writes before the keyed range by default, or after it with `after: true`; it may omit `rangeKey` only when the target file is empty. Successful writes return the new `fileHash`.

`Read Lines` normally requires the expected current `fileHash`, then returns LLM-readable text by default using a Mustache template with scalar metadata and a `{{#lines}}...{{/lines}}` section for repeated line rows. Pass `dangerous: true` instead of `fileHash` for unstable files that are being appended while you read. Prefer the smallest useful range; when you know marker text, use `Find` first to get line numbers, then read around those anchors. For large files, scan a narrow or estimated range with a compact custom `format`, then read the exact target range before editing. Use `{{content}}` for raw selected line text, or `{{#lines}}{{lineNumber}}:{{line}}` plus a newline before `{{/lines}}` for numbered rows. Empty results use `startLine: -1` and `endLine: -1`; `{{#rangeKey}}...{{/rangeKey}}` renders only when a range key is available. The `rangeKey` is an opaque base62 token and edits the entire returned range; read the exact subset you intend to modify before calling `Write Lines`. Pass `output: "json"` when a structured `lines[]` array is required. Pull-based Read Lines sends selected line content in `chunk.content`; the terminal `end` event carries the effective line range and file metadata top-level and does not repeat content.

`Write Lines` uses the `rangeKey` returned by `Read Lines` for `replace`, `insert`, and `delete`, so those edits target exactly the entire range that was read and fail if the file or range changed. If you only want a subset, read that exact subset first. `insert` writes before the keyed range by default, or after it with `after: true`; it may omit `rangeKey` only when the target file is empty. Successful line edits return the new `fileHash` and preserve the target file's existing trailing newline state.

`Find` returns regex matches with `path`, `startLine`, `endLine`, `startOffset`, `endOffset`, `fileHash`, and full matching-line `text`. `endLine` is inclusive; `endOffset` is exclusive. Use it to locate anchors before calling `Read Lines` for a small surrounding range or byte-oriented `Read` for an exact offset range. `filter` is an optional file-name glob matched against each file's base name, not the full relative path, and supports common brace alternatives such as `*.{js,ts,html}`. Find returns up to `limit` matches, defaulting to 100.
