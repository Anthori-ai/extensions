# Filesystem

First-party project extension that exposes filesystem controls through the sandboxed executable host.

`Create` returns `fileHash` for created files so callers can immediately make guarded `Read`, `Read Lines`, `Write`, or `Write Lines` calls. Directory creation returns an empty `fileHash`. `Info` returns file or directory metadata; for files, it includes the same opaque file version token.

`Read` normally requires the expected current `fileHash`, then returns decoded file content for a byte range selected by `offset` and `limit`, plus `fileByteCount`, `bytesRead`, the current `fileHash`, and an optional `byteRangeKey` formatted as `bytes:startOffset-endOffset:fingerprint` when bytes are returned. The visible offsets show the exact returned range, but callers must not edit or invent keys. Get the initial `fileHash` from `Create` or `Info`, then use the latest `fileHash` returned by creates, reads, and writes. For files that change too quickly to hash first, such as active logs, pass `dangerous: true` instead of `fileHash` to bypass stale-read detection. Pull-based Read sends file data in `chunk.content`; the terminal `end` event carries the read summary fields top-level and does not repeat content.

`Write` uses `byteRangeKey` values returned by `Read` for `replace`, `insert`, and `delete`, so raw byte edits target exactly the read range and fail if that range changed. `insert` writes before the entire keyed range by default, or after it with `after: true`; if you want a different anchor, read that exact byte range first. It may omit `byteRangeKey` only when the target file is empty. Successful writes return the new `fileHash`.

`Read Lines` normally requires the expected current `fileHash`, then returns LLM-readable text by default using a Mustache template with scalar metadata and a `{{#lines}}...{{/lines}}` section for repeated line rows. Pass `dangerous: true` instead of `fileHash` for unstable files that are being appended while you read. Prefer the smallest useful range; when you know marker text, use `Find` first to get line numbers, then read around those anchors. For large files, scan a narrow or estimated range with a compact custom `format`, then read the exact target range before editing. Use `{{content}}` for raw selected line text, or `{{#lines}}{{lineNumber}}:{{line}}` plus a newline before `{{/lines}}` for numbered rows. Empty results use `startLine: -1` and `endLine: -1`; `{{#lineRangeKey}}...{{/lineRangeKey}}` renders only when a lineRangeKey is available. The `lineRangeKey` is formatted as `lines:startLine-endLine:fingerprint`; the visible line numbers show the exact returned range, but callers must not edit or invent keys. Read the exact subset you intend to modify before calling `Write Lines`. Pass `output: "json"` when a structured `lines[]` array is required. Pull-based Read Lines sends selected line content in `chunk.content`; the terminal `end` event carries the effective line range and file metadata top-level and does not repeat content.

`Write Lines` uses the `lineRangeKey` returned by `Read Lines` or `Read Outline` for `replace`, `insert`, and `delete`, so those edits target exactly the entire range that was read and fail if the file or range changed. If you only want a subset, read that exact subset first. `insert` writes before the entire keyed range by default, or after it with `after: true`; if you want a different anchor, read that exact line range first. It may omit `lineRangeKey` only when the target file is empty. Successful line edits return the new `fileHash` and preserve the target file's existing trailing newline state.

`Find` returns regex matches with `path`, `startLine`, `endLine`, `startOffset`, `endOffset`, `fileHash`, and full matching-line `text`. `endLine` is inclusive; `endOffset` is exclusive. Use it to locate anchors before calling `Read Lines` for a small surrounding range or byte-oriented `Read` for an exact offset range. `filter` is an optional file-name glob matched against each file's base name, not the full relative path, and supports common brace alternatives such as `*.{js,ts,html}`. Find returns up to `limit` matches, defaulting to 100.

`Patch` applies raw Anthori patch text. Use `~Add: path`, `~Delete: path`, `~Update: path`, or `~Move: from-path -> to-path` sections. `~Move` can move only, or move and modify when followed by `@@` chunks. Add-file content lines start with `+`; update/move chunks start with `@@` and use space, `-`, and `+` line prefixes. Paths come from patch headers. Each update/move chunk must include enough context or removed lines to match exactly one location. Successful patches return `Done!`; invalid syntax, missing context, or ambiguous multiple matches fail the call. For compatibility, the parser also accepts OpenAI-style `*** Begin Patch` / `*** End Patch` wrappers and `*** Add File:`, `*** Delete File:`, `*** Update File:`, and `*** Move to:` headers.

`System > Filesystem > Shell` contains shell-shaped wrappers for common filesystem commands: `rg`, `ls`, `find`, `sed`, `nl`, `cat`, `head`, `tail`, `wc`, `stat`, `exists`, `mkdir`, `touch`, `rm`, `cp`, `mv`, `chmod`, and `chown`. Command-shaped controls accept one command string, either just the args or the full command name plus args, and return command stdout text in the closest practical style for the supported flags. The `rg` wrapper also supports `rg --files`. They call sandboxed `host.fs` operations instead of executing a shell. A single output pipe to `head`, `head -n N`, `tail`, or `tail -n N` is supported for familiar output trimming. Other shell operators, redirection, and unsupported flags fail the control call.
